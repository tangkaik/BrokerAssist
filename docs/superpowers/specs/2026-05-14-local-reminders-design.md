# 本地提醒通知设计

## 目标

构建第一版 CRM 提醒通知能力，通过手机本地系统通知帮助销售人员及时看到需要处理的客户事项，同时避免按客户逐条推送造成打扰。

## 产品范围

第一版使用移动端 App 本地排程通知，不接入服务端推送，不需要设备 token、APNs 或 FCM。

App 会按行业启用预定义提醒规则。第一版不开放给用户自定义提醒规则和推送时间。

提醒体验包含三个入口和展示面：

1. 首页「今日提醒」卡片。
2. 「今日提醒」详情页。
3. 客户列表里的提醒徽标，作为辅助浏览提示。

## 提醒规则

所有本地系统通知统一在用户手机所在时区的 09:00 推送。

保险经纪用户：

- 生日关怀：客户生日提前 3 天提醒。
- 节日礼品准备：节日前 7 天、3 天、1 天提醒。
- 保单缴费跟进：识别到明确缴费日期时，提前 3 天提醒。

通用 CRM 用户：

- 生日关怀：客户生日提前 3 天提醒。
- 节日关怀：节日前 7 天、3 天、1 天提醒。
- 久未联系跟进可以在后续数据质量足够后再加入。

房产顾问用户：

- 生日关怀：客户生日提前 3 天提醒。
- 节日关怀：节日前 7 天、3 天、1 天提醒。
- 看房或回访节点提醒等到后续有明确预约字段后再加入。

## 保单缴费日期识别

第一版不新增完整保单管理模块。

保单缴费日期只从现有客户信息中识别：

- 客户画像。
- 已保存的客户建议。
- 沟通记录。
- 如果图片识别结果已经进入记录数据，也可以作为识别来源。

识别逻辑必须保守。只有文本里出现明确缴费日、续费日或到期日时，才生成保单缴费提醒。用户没有输入明确日期时，不生成提醒。

可以生成提醒的例子：

- 「保单缴费日是 2026-06-01」
- 「6月1日前续费」
- 「下次保费 2026/06/01 到期」

不能生成提醒的例子：

- 「客户担心缴费压力」
- 「之前买过一份寿险」
- 「年缴保费 2 万」

## 通知合并

App 不能按客户逐条推送通知。提醒需要按日期和提醒类型合并。

同一天同一类提醒最多在 09:00 推送一条本地通知：

- 「今天有 3 位客户生日即将到来」
- 「今天有 10 位客户保单缴费需跟进」
- 「端午节还有 7 天，建议准备重点客户礼品」

用户点击通知后进入 App 的「今日提醒」详情页。如果通知来自某个具体类型，详情页可以尽量定位到对应分组。

## 首页设计

首页新增一个紧凑的「今日提醒」卡片。

卡片最多展示 3 条提醒预览。提醒文本源需要兼容 Markdown 风格的强调语法，方便后续使用加粗标签，而不必再做一套新的渲染逻辑。

示例预览：

- 「**生日关怀** · 3 位客户生日还有 3 天」
- 「**保单缴费** · 10 位客户需在 3 天内跟进」
- 「**节日礼品** · 端午节还有 7 天」

卡片包含清晰的「查看全部」入口。点击卡片进入「今日提醒」详情页。

如果今天没有提醒，卡片不消失，而是展示安静的空状态。这样首页布局更稳定，也能让用户知道提醒功能存在。

## 今日提醒详情页

新增「今日提醒」详情页。这个页面是处理提醒的主流程。

页面按类型分组展示提醒：

- 生日关怀
- 保单缴费
- 节日礼品
- 后续其他行业预定义提醒

每条客户提醒展示：

- 客户头像或姓名首字。
- 客户姓名。
- 提醒原因。
- 时间标签，例如「还有 3 天」或「今天处理」。
- 「查看客户」操作。
- 明确的「完成」操作。

点击「查看客户」进入对应客户详情页。从客户详情页返回时，需要回到「今日提醒」详情页，并保留用户之前在提醒列表里的位置。

只是打开客户详情，不代表提醒已经处理。只有用户点击明确的「完成」操作后，这条提醒才在当天消失。

已完成提醒可以从主列表隐藏，也可以用弱化样式展示在已完成状态里。第一版可以使用本地完成状态存储，键由用户、日期、提醒类型和目标对象 id 组成。

节日提醒不绑定单个客户。节日提醒行可以展示节日名称、提前天数和一个通用「完成」操作。

## 客户列表徽标

客户列表中，如果某个客户今天有待处理提醒，则在客户行旁边展示小徽标。

示例徽标：

- 「生日」
- 「缴费」
- 「节日」

徽标只是浏览客户时的辅助提示，不替代「今日提醒」详情页，也不承担完成逻辑。点击徽标可以进入提醒详情页或客户详情页，但提醒是否完成仍然需要显式点击「完成」。

## 本地通知排程

App 启动时，以及相关数据发生变化后，刷新本地提醒排程。

相关数据变化包括：

- 创建或编辑客户。
- 客户生日变化。
- 创建、编辑或删除沟通记录。
- 图片识别结果更新。
- 行业设置变化。
- 用户登录或退出。

排程器会计算未来一段时间内的提醒。第一版使用较小且实用的窗口，例如未来 30 天，避免触碰系统通知排程数量限制。

排程器在重新计算前，先取消当前用户之前排过的提醒通知，再写入新的提醒通知。完成状态需要阻止当天已完成提醒重新出现在首页卡片、提醒详情页和客户列表徽标中。

## 数据模型

第一版可以在 App 本地根据现有 API 数据计算提醒，不要求后端新增提醒表。

移动端需要新增本地提醒模型，表达：

- 提醒类型。
- 提醒发生日期。
- 关联客户 id，如果有。
- 标题和正文。
- 分组标题。
- 完成状态。
- 调试用来源信息，例如 `birthday`、`festival`、`payment_date_detected`。

提醒模型需要和 UI 组件分离。这样以后做服务端推送时，后端也可以生成同样形状的提醒数据，App 的首页卡片、提醒详情页和客户列表徽标可以继续复用。

## 异常处理

如果用户拒绝通知权限，首页卡片和提醒详情页仍然可以在 App 内正常使用。App 可以用轻量方式提示用户开启通知权限，但不能阻塞提醒列表。

如果本地通知排程失败，App 不能崩溃。可以记录日志或展示轻量提示，并继续保留 App 内提醒。

如果保单缴费日期识别不确定，就跳过提醒。第一版宁可漏提醒，也不要误提醒。

## 测试

单元测试需要覆盖：

- 生日提醒日期计算，包括跨月和跨年。
- 节日前 7 天、3 天、1 天提醒生成。
- 保守的保单缴费日期识别。
- 按日期和类型合并提醒。
- 完成状态隐藏当天提醒。
- 按行业选择预定义规则。

组件测试需要覆盖：

- 首页提醒卡片最多渲染 3 条预览。
- 今日提醒详情页按分组展示活跃提醒。
- 完成提醒后，提醒离开活跃列表。
- 客户列表为有提醒的客户展示徽标。

手动验证需要覆盖：

- 通知权限请求。
- App 未打开时，已排程的本地通知能出现在手机系统里。
- 点击系统通知后进入「今日提醒」详情页。
- 从客户详情页返回后，仍回到「今日提醒」详情页。

## 后续服务端推送路径

第一版明确使用本地通知。后续可以把提醒规则计算或通知投递迁移到后端，再接入 APNs/FCM。

为了保留这条路径，本地提醒类型和来源信息需要使用稳定 key。未来后端可以生成同样的提醒 occurrence 数据，由服务端负责推送，而 App 继续使用同一套首页卡片、提醒详情页和客户列表徽标 UI。
