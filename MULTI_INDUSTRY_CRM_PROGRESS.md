# 多行业轻量 CRM 改造进度

更新时间：2026-05-05

## 当前分支

- 当前开发分支：`codex/multi-industry-crm`
- 分支目的：在原保险经纪人助手基础上，扩展为可多行业适配的轻量级 CRM。
- 已同步到远端的最近提交：
  - `fdb08ab feat: add new user mobile home state`
  - `4396cd2 feat: add explicit industry onboarding`
- 当前还有未提交改动，主要集中在移动端记录流程、首页拆分、客户搜索和后续页面体验优化。

## 当前产品方向共识

- 不是另起一个全新项目，而是在当前项目分支上继续扩展。
- 当前重点不是先重构成复杂行业平台，而是把核心 CRM 流程做顺：
  - 新用户进来选择行业，默认通用。
  - 用户后续可在“我的”页面修改行业。
  - 首页、新建记录、创建客户、挂载已有客户这几个主流程先跑通。
  - 行业差异先主要体现在提示词、首页文案、AI 画像组织方式和少量字段建议上。
- JSON/文本字段暂时继续保留，用来承载不同业务里的非固定信息，避免过早把行业字段写死在数据库表结构里。
- MVP 阶段 AI 入口先收敛：
  - 客户详情页只保留结构化 AI 输出：`客户画像摘要` 和 `下一步建议`。
  - 客户详情页暂不提供“向 AI 问该客户情况”的自由问答入口，避免与画像/建议重复、增加认知负担。
  - 自由问答统一放到底部主导航的“AI 助手”页面。
  - 未来当客户记录很长、图片识别内容很多、需要围绕单个客户深问时，再恢复“客户深度问答/客户对话”入口。

- 改成录音机式 UI，而不是重复说明式 UI。
- 状态文案：

  - 录音中：`听写中`

- 转写仍调用讯飞接口，结果填入文本框。

### 7. 客户详情页 AI 入口收敛

### 8. AI 助手客户查询架构

- 客户筛选/统计类问题改为 Query Planner 架构：
  - 用户自然语言问题先交给 LLM 解析为白名单 JSON 查询计划。
  - 后端校验 `action`、`filters`、`sort`、`limit` 等字段。
  - 后端执行确定性过滤和计数，并输出稳定的客户链接格式。
- 已知稳定的简单问题保留为前置快路径，例如“列出所有客户”“两个月没联系的客户”“优先跟进客户”，避免不必要的 LLM 调用。
- Query Planner 的入口不再依赖固定城区名单，只判断是否像客户筛选/统计问题；非客户查询由规划器返回 `action: none` 或直接走普通全局问答。
- 目标是让 LLM 做语义理解，不让 LLM 直接遍历客户、计数或决定最终客户集合。
- 这类问题不再需要把全量客户数据拼进 prompt；LLM 只看到用户问题和可查询字段 schema。
- 当前支持的规划字段：
  - `gender`
  - `location`
  - `age`
  - `tags`
  - `keywords`
  - `stale_contact_months`
- 当前年龄还没有独立数据库列，执行器会从标签和摘要中的“xx岁”提取。后续建议补正式 `age` 字段。
- 已验证示例：
  - `有多少住在海淀区的女客户`
  - `列出海淀区的客户`
  - `徐汇区的客户有哪些`
  - `55岁以上女性客户有哪些`

### 9. AI 助手 MVP 稳定性改进

- 移动端全局 AI 请求会携带最近 8 轮对话上下文，供后端支持轻量追问；Web 全局聊天同步携带最近 16 条消息。
- 全局 AI 兜底路径加入 prompt 客户数保护：最多放入最近活跃的 50 位客户，避免客户量增长后 token 失控。
- 客户专属问答不再强制要求客户画像已生成；如果画像未 ready，会降级为仅基于最近 3 条记录回答，并明确提示信息边界。
- 图片问答/图片识别失败时，移动端和 Web 会给出可继续操作的文本提示，而不是只显示服务错误。
- 暂缓项：SSE 流式输出、KimiClient 连接池复用、服务端对话历史持久化、动态推荐问题。

### 10. 三行业 Prompt 隔离与样例数据

- 当前 MVP 行业范围收敛为：
  - 通用
  - 保险经纪
  - 房产顾问
- 行业是用户级设置，不在客户表上保存行业字段；首次选择后锁定，移动端不提供修改入口。
- 后端已增加 `users.industry_key`，移动端行业选择会同步保存到后端。
- 客户画像、下一步建议、客户问答、全局 AI 问答、Query Planner 都会按当前用户行业注入不同角色和关注维度。
- 已新增轻量行业配置：
  - `server/app/core/industry_profiles.py`
- 已扩展测试数据导入脚本：
  - `server/import_customers.py --industry insurance --yes`
  - `server/import_customers.py --industry real_estate --yes`
  - `server/import_customers.py --industry mixed --yes`
  - 不传 `--yes` 时只提示风险，不会清空数据。
- 房产样例覆盖：首套刚需、改善三居、学区关注、投资型、卖一买一、婚房、租转买、品质改善等场景。

## 已发现但当前 MVP 暂不解决

这些问题是真问题，但目前不阻塞 MVP 验证。建议标记为“上线前必须收口”或“客户量增长后处理”。

### 1. 安全与稳定性

- `server/app/services/xunfei_service.py` 中 SSL 校验禁用：`verify=False`
  - 当前判断：内测阶段可暂缓。
  - 未来动作：正式上线前恢复 SSL 校验，并处理证书兼容问题。
- `server/app/core/config.py` 中 CORS 默认全开放：`cors_allow_origins="*"`
  - 当前判断：内测调试方便，暂缓。
  - 未来动作：正式部署时改为明确域名白名单。
- `server/app/core/config.py` 中默认 JWT 密钥硬编码。
  - 当前判断：如果生产环境已由 `.env` 覆盖，当前风险可控，但不应长期保留。
  - 未来动作：移除危险默认值，启动时强制校验生产密钥。
- `server/app/services/vision_service.py` 中 vision API 无超时。
  - 当前判断：稳定性风险，不是当前主流程阻塞项。
  - 未来动作：增加合理 timeout、错误提示和重试/降级策略。

### 2. 客户搜索正式版

- 当前拼音搜索是前端轻量版，适合客户量小的时候。
- 未来客户量增大后，后端应增加：
  - `name_pinyin`
  - `name_initials`
- 并建立索引，让姓名、电话、标签、全拼、首字母统一走后端搜索。
- 这项已同步记录在 `IMPROVEMENT_PLAN.md`。

### 3. 文件导入

- 暂不在记录输入框里加“文件上传”按钮。
- 原因：
  - 移动端主动找文件不如 PC 顺手。
  - 文件更多来自微信、邮件、网盘或系统文件 App。
- 未来更合理的方向：
  - 支持从微信/邮件/文件 App 使用系统分享导入到 CRM。
  - 而不是优先在 CRM 内部放一个文件选择按钮。

### 4. 数据结构

- 目前仍保留 JSON/文本字段，用于承载跨行业的灵活信息。
- 暂不急于为每个行业建立大量专用表字段。
- 未来方向：
  - 稳定通用实体继续结构化，例如客户、记录、任务、标签、行业、用户配置。
  - 行业差异内容继续用 JSONB/文本承载。
  - 当某些字段跨多个行业稳定复用，再提升为正式列。

5. 后续再集中处理上线前安全、稳定性和后端搜索索引。
